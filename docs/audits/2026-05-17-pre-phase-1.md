## 5. Glossary — PASS (with one borderline term)

Ubiquitous-language glossary at `docs/02-classes.md:250–265` covers:
Prompt, Intent, Parameter, PrimaryFeature, Modifier, Adapter,
AdapterRefusal, Extras, Run, Run-id, Trace, Critique, SanityWarning,
ContractViolation.

Class/data terms used in `02-classes.md` that are NOT in the glossary:

- `SanityResult`, `DimensionMismatch`, `SanityChecker`, `PlanResult`,
  `ExecutionResult`, `RunConfig`, `Tokens`, `ModelPrice`,
  `PromptsBundle` (this last appears as a parameter type at
  `docs/02-classes.md:122`).

These are mostly infrastructure types (return/parameter records, config
dataclasses) rather than domain terms appearing in the planner system
prompt or user-facing docs — `SanityResult` is paired with the
glossed `SanityWarning`; `ContractViolation` is glossed and stands in
for the per-kind validation result family; `Run`/`Run-id` cover the
`RunConfig`/`ExecutionResult` cluster. No domain-language gap that
would surface in code review or prompt engineering for phase-1.

One borderline term: `PromptsBundle` (`docs/02-classes.md:122`) is used
in a public signature without a definition anywhere. Recommend either
(a) define it in the glossary (it's a domain concept: "the versioned
collection of planner system prompt + few-shots, hashed for caching
diagnostics per ADR-0003"), or (b) drop the type name in favour of a
plainer parameter (e.g., `prompts_dir: Path`). Borderline rather than
hard FAIL because phase-1 does not implement `Planner.plan` — that's
phase-2a, where this will be code-visible.
