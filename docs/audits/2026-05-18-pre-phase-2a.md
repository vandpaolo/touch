## 5. Glossary — FAIL

The ubiquitous-language glossary (`docs/02-classes.md:250–280`) defines
29 terms. The following identifiers appear in `docs/02-classes.md`
(class names / method signatures / DDD analysis) but are not in the
glossary:

- **`Dimension`** (classes.md:139) — return type of
  `SanityChecker._extract_dimensions(prompt: str) -> list[Dimension]`.
  Glossary has `DimensionMismatch` but not `Dimension`. Phase 2a's
  minimum deliverable explicitly ships a `Dimension` value object
  (phase-2a.md:94), so this is a real domain term about to enter the
  code. Should be added.
- **`Refiner`** (classes.md:271) — referenced inside the `Loop`
  glossary entry: "state machine across Planner → SanityChecker →
  Worker → Executor (→ Evaluator → Refiner in v0.1)". `Refiner` is
  introduced as a named role but has no definition. Either drop the
  word from the Loop entry or add a stub `Refiner` glossary row marked
  v0.1.
- **`Mesh`** (classes.md:192) — return type of
  `Render._load_step(path: Path) -> Mesh`. This is a PyVista-native
  type, not a domain term, so it's borderline; flagging only because
  the audit asks for completeness. A one-line clarification ("PyVista
  mesh handle, not a domain term") inside the Render diagram or the
  glossary would close the ambiguity.

The first two are clear gaps. The third is ambiguity rather than a
gap. Recommend a glossary touch-up before phase-2a-end (the
[`phase-1-report.md`](../phases/phase-1-report.md) recommendation #8
glossary follow-up is the natural home for this work — phase-2a.md:254
explicitly defers it).
