## 11. T0-specific: risk register sanity — FAIL

`phase-T0.md:96-108` covers R1–R9: PyInstaller+OCP, sidecar spawn on
Windows, WS connect race, port collision, three.js per-face-ID mesh,
Windows VM iteration cost, SmartScreen on unsigned installers, CI vs
local VM build, and software-GL in VMs.

R7 covers *raw* SmartScreen (unsigned installer warning) — good. But
the following adversarial failure modes for a Python + native-CAD-
kernel Windows installer are **not** in the table:

- **OCP / OCCT redistribution + licensing.** OpenCascade ships under
  LGPL-2.1 with an exception; redistributing the native libs inside
  a Windows installer carries documented obligations (notice file,
  source availability for the LGPL'd portion, no static linking
  pitfalls). build123d itself is permissively licensed but it bundles
  OCP. Missing as a risk — should be at least a noted obligation,
  even if mitigation is "include LGPL notice + link to OCP source."
- **Antivirus false-positives on unsigned PyInstaller bundles.**
  PyInstaller-bundled Python executables are a well-known
  AV/EDR false-positive trigger (the Python interpreter + ctypes +
  the bootloader stub matches multiple heuristic signatures).
  Distinct from SmartScreen (R7) — AV/EDR may quarantine the
  installer or the sidecar binary outright. The Day-6 VM clean-
  install verification should explicitly include "Windows Defender
  scan does not quarantine" as an exit check.
- **OCP wheel Python-version pin vs PyInstaller's bundled Python.**
  OCP wheels are tightly pinned to specific Python minor versions
  (e.g. cp312-cp312-win_amd64). The PyInstaller spec must use the
  *exact same* Python minor as the wheel — a mismatch produces a
  cryptic import failure inside the bundle, not the venv. Not called
  out as a risk distinct from R1 (which talks about binary
  collection, not Python-ABI pinning).
- **Windows Defender SmartScreen reputation (separate from raw
  signing).** R7 covers unsigned. But even a *signed* installer
  from a brand-new code-signing cert with no reputation will trip
  SmartScreen ("Windows protected your PC") until enough installs
  have accumulated, or an EV cert is used. Worth a one-line note
  that T13's signing work doesn't immediately eliminate this — only
  EV signing or accumulated reputation does. (Not material to T0
  itself, but worth noting in the T0 report's "known issues to flag
  for T13" so T13 doesn't assume signing = no warning.)
- **electron-builder NSIS + per-user vs per-machine install.** A
  non-admin friend (`phase-T0.md:30-31` says non-admin) needs a
  per-user install. NSIS's `oneClick: true` + `perMachine: false`
  is the default for electron-builder, but a misconfiguration drops
  an admin prompt — explicit risk worth listing.
- **Sidecar `--onedir` resource size + Defender scan time.** The
  PyInstaller `--onedir` build of OCP-bearing Python is on the order
  of 300–500 MB unpacked; Defender's real-time scan on first launch
  can stall the sidecar process visibly (cold-start latency). Not
  in R1/R4; partial overlap with R4 in `01-requirements.md` but not
  carried into phase-T0.

R1 partially covers the first three at the surface level ("PyInstaller
+ OCP native libs"), but each of these is a *distinct* failure mode
that deserves its own mitigation. Recommend adding at minimum
**R10 (OCP/LGPL redistribution), R11 (AV false-positives, including
the Defender-scan-quarantine exit check on Day 6), R12 (OCP wheel
Python-version pinning explicit in the spec), R13 (NSIS perMachine vs
perUser explicit setting)** before the phase greenlights.
